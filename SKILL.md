## 核心工作流程

当你收到用户的想法时，按照以下流程处理：

### 第零步：创建项目文档

**创建项目记录文件：**

**文件维护原则：**

1. **实时更新**：每次完成一个步骤、遇到问题、做出决策后，立即更新 PROJECT.md
2. **时间戳**：每次更新都要更新「最后更新时间」
3. **可读性**：保持文档结构清晰，便于快速查阅
4. **完整性**：记录所有重要信息，包括决策、变更、问题等
5. **持久化**：确保文件保存到磁盘，下次启动时可以继续

**文件位置优先级：**
1. 用户明确指定的项目目录
2. 当前工作目录
3. 用户主目录下以项目名命名的文件夹

**使用场景：**
- 项目中断后继续
- 向用户汇报进度
- 回顾项目历史
- 团队共享项目信息

### 第一步：读取历史经验

**在开始新项目之前，首先查看经验文件夹：**

经验文件夹路径：`./1052/`（技能目录下的 1052 文件夹）

**读取流程：**
1. 扫描 1052 文件夹中的所有 .md 文件
2. 按修改时间排序（最新的在前）
3. 读取每个经验文件的摘要（标题 + 核心要点）
4. 将相关经验呈现给用户参考

**经验文件命名规范：**
```
[日期]_[经验主题].md
示例：2024-03-13_Excel自动化最佳实践.md
```

**经验文件结构：**
```markdown

**注意事项：**
- 如果 1052 文件夹为空或不存在，跳过此步骤
- 只展示摘要，不一次性展示全部详细内容
- 让用户选择是否深入查看某个经验
- 可以将相关经验应用到当前项目中

**如果发现与当前项目高度相关的经验：**
- 在需求分析和方案设计阶段参考这些经验
- 将相关经验应用到执行计划中
- 在 PROJECT.md 中引用参考的经验

#### 2.1 识别想法类型

首先判断用户想法属于以下哪种类型：

| 类型 | 特征 | 示例 |
|------|------|------|
| **功能需求** | 用户想要实现某个功能 | 「我想做一个文件加密工具」 |
| **问题解决** | 用户遇到问题需要解决 | 「我的网站加载很慢，怎么优化」 |
| **学习探索** | 用户想了解或学习某事物 | 「我想学 Python 爬虫」 |
| **项目规划** | 用户有较大的项目想法 | 「我想做一个电商平台」 |
| **工具改进** | 用户想改进现有工具 | 「能不能把 Excel 自动化」 |

#### 2.2 提取关键信息

从用户输入中提取以下要素：

**Who（谁）：**
- 目标用户是谁
- 谁会使用这个成果
- 是否有特定的用户群体

**What（做什么）：**
- 核心功能是什么
- 主要输出是什么
- 期望达到什么效果

**Why（为什么）：**
- 用户的动机和目的
- 解决什么问题
- 带来什么价值

**Where（在哪里使用）：**
- 使用场景是什么
- 部署环境
- 运行平台

**When（何时使用）：**
- 时间要求
- 紧急程度
- 使用频率

**How（如何实现）：**
- 技术偏好
- 预算约束
- 质量标准

#### 2.3 建立理解确认

使用以下方式确认理解正确：

**复述理解：**
```
根据我的理解，你想要[核心需求]，目的是[预期目标]，对吗？
```

**补充确认：**
```
我想确认几点：
1. [确认点1]
2. [确认点2]
3. [确认点3]
```

### 第三步：想法扩展与分析

#### 2.1 补全遗漏信息

分析用户想法中可能缺失的部分并补充：

**功能层面：**
- 核心功能（必须有的）
- 辅助功能（最好有的）
- 扩展功能（可以有的）

**非功能层面：**
- 性能要求（速度、并发）
- 安全性要求
- 可用性要求（易用性）
- 可维护性要求

#### 2.2 识别潜在问题

| 类别 | 可能的问题 | 应对策略 |
|------|------------|----------|
| 技术难度 | 技术栈过于复杂 | 建议简化方案或分阶段实现 |
| 资源需求 | 需要额外的工具或服务 | 评估替代方案 |
| 依赖关系 | 依赖第三方服务 | 评估可靠性和成本 |
| 边界情况 | 未考虑异常情况 | 补充错误处理 |
| 扩展性 | 未考虑未来扩展 | 预留扩展接口 |

#### 2.3 提供专业建议

**优化建议：**
- 推荐更好的技术方案
- 建议更高效的实现方式
- 提供行业最佳实践

**风险提示：**
- 指出潜在风险
- 建议预防措施
- 提供备选方案

#### 3.2 步骤细化方法

**分解原则：**
- 每个步骤应该在 1-2 小时内可完成
- 每个步骤都有明确的输入和输出
- 步骤之间有清晰的依赖关系

**细化模板：**
```
### 步骤 X：[步骤名称]

**目标：** [要达成的具体目标]

**输入：** [需要的资源或前置条件]

**输出：** [产出的成果]

**具体操作：**
1. [操作1]
2. [操作2]
3. [操作3]

**检查点：**
- [ ] 验证点1
- [ ] 验证点2

**可能遇到的问题：**
- 问题1及解决方案
- 问题2及解决方案
```

#### 3.3 依赖关系分析

**识别依赖：**
- 哪些步骤依赖其他步骤
- 哪些步骤可以并行执行
- 哪些步骤有外部依赖

**并行优化：**
- 将独立的步骤并行安排
- 减少总完成时间

#### 3.4 风险评估

为每个阶段评估风险：

| 阶段 | 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|------|----------|
| 准备 | 依赖安装失败 | 中 | 中 | 准备替代方案 |
| 实现 | 技术难点 | 高 | 高 | 预留缓冲时间 |
| 测试 | 环境问题 | 低 | 高 | 提前准备环境 |

## 实现原则

### 主动扩展
- 不要等待用户给出完整需求，主动分析和补充
- 预见潜在问题并提前规划解决方案
- 提供专业建议，优化用户想法

### 渐进明细
- 初期给出宏观计划，细节逐步完善
- 每个步骤都可独立验证和调整
- 保持灵活性，根据实际情况调整计划

### 用户参与
- 关键决策点需要用户确认
- 定期同步进度，保持透明
- 重视用户的反馈和调整需求

## 完整示例

### 技能处理

```
